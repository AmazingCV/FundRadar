# 开发规范

## 基本原则

当前不需要第二仓库。所有稳定版本都推送到：

```text
https://github.com/AmazingCV/FundRadar
```

原因：

- 单仓库更容易维护；
- commit 和 tag 已经能管理版本；
- 分支可以管理实验功能；
- 不需要重复推送第二仓库，避免版本混乱。

## 修改前检查

每次功能修改前先检查工作区：

```bash
git status
```

确认当前分支、未提交文件和本次修改范围。

## 测试要求

修改完成后必须运行测试。当前项目可用方式为：

```bash
python -c "import sys,pathlib; sys.path.insert(0, str(pathlib.Path('.deps').resolve())); sys.path.insert(0, str(pathlib.Path('src').resolve())); import pytest; raise SystemExit(pytest.main(['-p','no:cacheprovider','tests']))"
```

如果未来环境已正常安装依赖，也可以直接运行：

```bash
python -m pytest
```

## 不提交生成物

提交前确认以下内容没有进入 Git：

- `.deps/`
- `data/cache/`
- `data/raw/`
- `data/snapshots/`
- `reports/`
- `*.xlsx`
- `*.csv`
- `*.pkl`
- `*.parquet`

检查命令：

```bash
git status
git ls-files
```

## 提交流程

稳定修改后提交并推送：

```bash
git add .
git status
git commit -m "<简短说明>"
git push
```

提交信息应说明实际修改内容，不要使用空泛描述。

## Token 安全

不要把 GitHub token 写进：

- remote URL；
- README；
- config；
- docs；
- 代码；
- 日志。

remote 应保持为普通 HTTPS 地址：

```text
https://github.com/AmazingCV/FundRadar.git
```

如果发现 remote URL 中出现 token，立刻改回：

```bash
git remote set-url origin https://github.com/AmazingCV/FundRadar.git
```

## 分支规范

后续开发新功能时建议使用分支：

```bash
git checkout -b feature/<功能名>
```

示例：

```bash
git checkout -b feature/v3-portfolio-simulator
```

开发完成、测试通过后再合并 `main`。不要直接在 `main` 上做大规模实验。
